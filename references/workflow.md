# 创作工作流程规范

本文件定义两阶段创作流程、README项目记忆系统、双语输出规则。
每次开始创作前必读。

---

## 一、语言模式判断

收到创作需求后，先判断语言模式：

| 用户描述关键词 | 语言模式 | 输出规则 |
|---|---|---|
| 英文 / 海外平台 / overseas / English / 面向海外 | 双语模式 | 见下方双语规则 |
| 无特殊说明 | 中文模式 | 全部中文输出 |

**双语模式输出规则：**
- 剧本基础信息表：中文
- 故事梗概：中文
- 人物小传、人物设定框架内容：中文
- 人名、地名：英文或适配目标文化的名字
- 台词：英文为主，保持真人英文呼吸感，禁止翻译腔
- 场景描述：中英双语（中文在上供校对，英文在下供生成）
- 运镜备注：中英双语（中文在上，英文在下）
- README.md：中文

**双语场景描述格式：**

```
【场景01·天台·夜·外景 / Scene 01·Rooftop·Night·Exterior】

（宽阔的天台，城市的灯火在远处低低连成一片。
  地面的防水层已经开裂，有几处积水反射着冷光。
  整个空间有一种高处的孤立感。AI复杂度：中等。）

(A wide rooftop. City lights stretch low across the horizon.
  The waterproof coating is cracked, puddles reflect cold light.
  The space feels isolated, suspended above everything.
  AI complexity: Medium.)

【运镜：宽全景建立空间→缓推至人物中景→情绪高点特写】
【Camera: Wide establishing shot → slow push to mid-shot → close-up at emotional peak】
```

**英文台词真人呼吸感规范：**
- 用缩写（I'm / don't / can't / gonna / wanna）
- 保留打断（用——）、停顿（用…或beat）、说到一半停住
- 禁止出现过于书面的完整句（"I would like to express my sincere apology"）
- 每个角色有自己的英文说话节奏，不可互换

## 二、两阶段创作流程

### 第一阶段：框架输出

按以下顺序输出，全部完成后暂停，等待用户确认：

1. 剧本基础信息表
2. 故事梗概（一句话 + 完整版）
3. 核心冲突线 + 情绪曲线设计
4. 全部人物设定（每人完整五板块）
5. 人物关系图谱
6. 分集大纲（全部集数）

**输出完成后，固定显示以下提示：**

---
框架输出完毕。请确认以下几点后告诉我是否需要调整：

1. 人物设定是否符合预期？（性格、形象、台词特点）
2. 故事走向是否正确？（梗概、冲突线、情绪曲线）
3. 分集大纲的节奏和钩子是否满意？

如需调整，直接告诉我哪里改、怎么改。
调整完毕确认后，告诉我「通过，开始写剧本」，进入正式剧本创作。
---

### 第二阶段：分集剧本创作

收到「通过，开始写剧本」或「继续写第X集」后启动。

**每次写剧本正文之前的强制流程：双 agent 审核机制**

收到「通过，开始写剧本」或「继续写第X集」后，
禁止直接输出剧本正文。必须先完成以下完整流程：

---

**STEP 1：主 agent 填写首帧描述框**

读取分集大纲中即将写的集数的「首帧画面」列，
按照 script-standards.md「5.1.1」的格式，
为即将写的每一集填写完整的「首帧描述框」。
填完后不得自查，直接进入 STEP 2。

---

**STEP 2：启动首帧审核子 agent（对立角色）**

主 agent 用 Agent 工具启动一个独立的审核子 agent，
传入以下完整 prompt：

「在开始审核之前，你必须先读取以下文件获取最新审核标准，
不得使用记忆中的任何旧版本标准：
~/.claude/skills/screenplay/references/script-standards.md
读取完成后，严格按照该文件的最新内容执行审核。

你是一个极度挑剔的短剧平台审片人，有十年抖音爆款审片经验。
你的KPI是拒稿，不是通过。
你默认每一个首帧描述都有问题，
你的任务是证明它不值得通过，
除非它让你找不到任何可以拒绝的理由。

审核标准（逐条检查，任何一条不通过即整体不通过）：
1. 第一帧画面是否在0秒内制造「这是什么/接下来怎样」的强烈问题？
   平静画面、独处画面、环境交代画面——全部不通过。
2. 前3秒是否完成了视觉冲击/台词冲击/悬念冲击中的至少一种？
   「他站在窗边」「她看着远方」——全部不通过。
3. 第一句台词是否是信息炸弹或情绪炸弹？
   任何废话开场——不通过。
4. 禁止清单六条是否全部为「无」？
   有任何一条为「有」——不通过。
5. 使用的开场模式是否有明确的主模式标注？
   必须有主模式，副模式可选。
   允许标注「主：X型，副：X型」的复合形式。
   三种全选、完全不选、或无法对应任何一种模式——不通过。
   同时满足两种模式并标注主副——通过。

审核结论格式：
通过：逐条说明为什么符合标准，语气可以勉强接受但不夸奖。
不通过：逐条说明哪里有问题，指出具体是哪一句或哪一帧的问题，
        不提供修改方案，只指出问题所在。

待审核内容：[主 agent 将填写好的首帧描述框完整内容插入此处]」

---

**STEP 3：根据首帧审核结果执行**

情况A——审核子 agent 返回「通过」：
主 agent 将描述框和审核通过结论一起输出给用户，
固定提示语：
「以下首帧描述已通过内部审核，请确认后
告诉我「首帧没问题，开始写」，我才开始写剧本正文。
如需调整，直接告诉我哪里改。」

情况B——审核子 agent 返回「不通过」：
主 agent 按照审核意见修改描述框，
修改完成后立即再次启动审核子 agent 重新审核（STEP 2）。
最多修改两次。

情况C——修改两次后仍不通过：
主 agent 将三个版本的描述框和审核子 agent 的
所有不通过理由，完整输出给用户，
固定提示语：「首帧描述经过两次修改仍存在争议，
请用户裁决后告诉我最终方案。」
由用户做最终决定。

---

**STEP 4：收到用户确认后写剧本正文**

收到用户「首帧没问题，开始写」之后，
主 agent 开始写剧本正文。
第一场戏的第一帧必须与用户确认过的描述框完全一致。

---

**STEP 5：正文完成后启动一致性校验子 agent**

每集正文写完后，在输出给用户之前，
主 agent 用 Agent 工具启动一致性校验子 agent，
传入以下完整 prompt：

「在开始校验之前，你必须先读取以下两个文件获取最新标准，
不得使用记忆中的任何旧版本标准：
~/.claude/skills/screenplay/references/script-standards.md
~/.claude/skills/screenplay/references/character-standards.md
读取完成后，严格按照这两个文件的最新内容执行校验。

你是一个严格的短剧剧本质检员。
你的任务是对比以下两份内容，找出所有不一致和质量问题。
你不负责夸奖，只负责列出问题清单。
没有问题时，输出「校验通过」。

检查项目一：首帧一致性
对比用户确认过的首帧描述框和剧本正文第一场戏，逐句检查：
- 第一帧画面是否和描述框完全一致？
- 第一句台词是否和描述框完全一致？
- 前3秒的情绪任务是否在正文里实际完成？
不一致的地方，引用正文原句和描述框原句对比列出。

检查项目二：台词AI腔检测
检查所有台词，找出以下问题：
- 任意两个角色的台词可以互换而不违和
- 有角色用超过3句话解释自己的感受或动机
- 出现「对不起」「我明白你的感受」「谢谢你」等整洁回应
- 一段对话里没有任何停顿、打断、说到一半停住的台词
有问题的，引用具体台词标注问题类型。

检查项目三：角色声音一致性
对照当前项目的人物设定文件，
检查每个出场人物的所有台词，
判断说话方式是否和「E. 台词特点」完全一致。

主 agent 在启动本校验子 agent 时，
必须同时传入以下内容：
1. 本集所有出场人物的「E. 台词特点」完整内容
   （从项目框架文件中提取，逐字传入，不得省略）
2. 首帧描述框（用户确认版）
3. 剧本正文完整内容

如果主 agent 未传入人物台词特点，
校验子 agent 必须拒绝开始校验，
返回：「缺少人物台词特点数据，请主 agent 补充后重新提交。」

待校验内容：
首帧描述框（用户确认版）：[主 agent 插入]
剧本正文：[主 agent 插入]」

---

**STEP 6：根据一致性校验结果执行**

情况A——校验子 agent 返回「校验通过」：
主 agent 将完整剧本正文输出给用户，
固定提示语：
「第X–X集输出完毕，已通过内部质检。请确认：
1. 两集之间的连贯性是否自然？
2. 台词呼吸感是否到位？
3. 集尾钩子是否足够强？
确认无误后告诉我「继续」，输出下两集。」

情况B——校验子 agent 返回有问题：
主 agent 按照问题清单修改剧本正文，
修改完成后再次启动校验子 agent 重新校验（STEP 5）。
最多修改两次。

情况C——修改两次后仍有问题：
主 agent 将当前版本剧本、校验子 agent 的问题清单，
完整输出给用户，固定提示语：
「剧本经过两次修改仍存在以下问题，请用户裁决：」
由用户决定是否接受或继续修改。

**每次输出2集，输出完毕后暂停，固定显示以下提示：**

---
第X–X集输出完毕。请确认：

1. 两集之间的连贯性是否自然？
2. 台词呼吸感是否到位？
3. 集尾钩子是否足够强？

如需调整，告诉我具体问题。确认无误后告诉我「继续」，输出下两集。
---

**每次续写前，先读取项目 README.md，确认：**
- 当前已完成集数
- 上次用户的调整意见
- 上两集的结尾钩子（保证接续自然）

---

## 三、README.md 项目记忆系统

### 3.1 创建时机

第一阶段框架通过、用户说「通过，开始写剧本」时，
在当前项目文件夹下创建 README.md。

### 3.2 README.md 模板

```markdown
# [剧名] · 项目状态文件

> 本文件由 screenplay skill 自动维护，记录项目当前状态。
> Claude Code 每次进入本项目时，请先读取本文件再开始工作。

## 基本信息

| 项目 | 内容 |
|---|---|
| 剧名 | |
| 类型 | |
| 总集数 | |
| 单集时长 | |
| 语言模式 | 中文 / 双语 |
| 创建日期 | |
| 最后更新 | |

## 一句话梗概

（填入）

## 人物清单

| 姓名 | 定位 | 一句话特征 |
|---|---|---|
| | | |

## 当前进度

- **阶段**：框架确认中 / 剧本创作中 / 已完成
- **已完成集数**：第 X 集 — 第 X 集
- **待创作集数**：第 X 集 — 第 X 集

## 框架状态

- 框架已通过用户确认：是 / 否
- 最后一次框架调整说明：（记录用户改了什么）

### 第1–2集
- 第1集核心事件：（一句话）
- 第2集核心事件：（一句话）
- 第2集结尾钩子：（一句话，供下次续写时接续用）

### 第3–4集
...

## 待解决问题 / 用户备注

（记录用户在过程中提出的待处理问题、特殊要求、风格偏好）
```

### 3.3 更新时机与内容

| 时机 | 更新内容 |
|---|---|
| 框架通过后 | 创建文件，填入基本信息、梗概、人物清单、框架状态 |
| 每完成2集 | 追加该2集的剧集摘要，更新当前进度和最后更新日期 |
| 用户提出调整 | 更新「待解决问题/用户备注」和「最后一次框架调整说明」 |
| 全部完成 | 将阶段改为「已完成」 |

### 3.4 下次调用时的行为（进入已有项目）

Claude Code 进入项目文件夹，检测到 README.md 存在时：

1. 读取 README.md 全部内容
2. 报告当前状态：「检测到项目进度，当前已完成第X集，待创作第X集起」
3. 询问用户：「是否直接继续写第X–X集，还是先调整框架？」
4. 按用户指令无缝接续，不重新开始
